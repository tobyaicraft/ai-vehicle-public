# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Bare-metal, multi-core embedded C project for the **Infineon AURIX TC275T** (TC27xD family) microcontroller. Three TriCore cores run independently, synchronized at startup. Currently a skeleton/template with empty main loops.

## Toolchain & IDE

- **IDE**: Infineon AURIX Development Studio (ADS) — Eclipse CDT-based
- **Compiler**: TASKING VX-toolset for TriCore (`cctc` compiler, `astc` assembler)
- **Debugger**: iSYSTEM winIDEA via Infineon DAS over USB
- **Build system**: GNU Make, auto-generated by ADS (do not hand-edit makefiles in `TriCore Debug (TASKING)/`)
- **C standard**: C99 with C++14 support, MISRA-C 2012 checking enabled
- A parallel GCC/GNUC toolchain path exists (linker script + compiler abstraction) but TASKING is the primary toolchain

## Build

Builds are managed through ADS IDE. The auto-generated makefiles live in `TriCore Debug (TASKING)/`:

```bash
# Command-line build (from project root, requires TASKING toolchain on PATH)
make -C "TriCore Debug (TASKING)" all
```

Output artifact: `TriCore Debug (TASKING)/ai-vehicle-public.elf` (also `.hex`)

Two build configurations exist: **TriCore Debug (TASKING)** and **TriCore Release (TASKING)**. Debug compiles only a minimal set of iLLD drivers; Release includes all peripheral drivers.

## Architecture

### Multi-Core Structure

| File | Entry Point | Role |
|------|-------------|------|
| `Cpu0_Main.c` | `core0_main()` | Primary core — disables CPU + Safety watchdogs, emits sync event |
| `Cpu1_Main.c` | `core1_main()` | Secondary — disables CPU watchdog, waits for sync |
| `Cpu2_Main.c` | `core2_main()` | Secondary — disables CPU watchdog, waits for sync |

All cores synchronize via `cpuSyncEvent` (defined in Cpu0_Main.c, externed in others) before entering their main loops. CStart code in `Libraries/iLLD/TC27D/Tricore/Cpu/CStart/` handles core initialization before these entry points.

### Key Directories

- **`Libraries/iLLD/TC27D/Tricore/`** — Infineon Low Level Drivers (iLLD). Each peripheral has `Std/` (register-level API) and higher-level driver subfolders. Do not modify these vendor-provided files.
- **`Libraries/Infra/`** — Compiler abstraction (`Platform/Tricore/Compilers/`) and SFR register definitions (`Sfr/TC27D/_Reg/`). Supports TASKING, GCC, GNUC, GHS, DCC toolchains.
- **`Libraries/Service/CpuGeneric/`** — System services (BSP, math, time, communication interfaces, standard interfaces).
- **`Configurations/`** — Project-level configuration:
  - `Ifx_Cfg.h` — Master config: 20 MHz crystal, 200 MHz PLL
  - `Debug/sync_on_halt.c` — TASKING debugger cache synchronization

### Linker Scripts

- `Lcf_Tasking_Tricore_Tc.lsl` — Primary (TASKING toolchain)
- `Lcf_Gnuc_Tricore_Tc.lsl` — Alternative (GCC toolchain)

Per-core memory: each core gets its own DSPR (112-120 KB) for stack, CSA (8 KB), and heap. Shared LMU RAM is 32 KB. Program flash is 2×2 MB.

### Memory Map Highlights

- Program Flash: `0x80000000` (4 MB total)
- Data Flash: `0xaf000000` (384 KB)
- CPU0 DSPR: `0x70000000` (112 KB), CPU1: `0x60000000` (120 KB), CPU2: `0x50000000` (120 KB)
- LMU RAM: `0x90000000` (32 KB)
- Interrupt vector table: `0x801F4000`

## Important Conventions

- The `Libraries/` directory is vendor-provided (Infineon iLLD SDK). Avoid modifying files there.
- When adding new peripherals to the Debug build, both include paths (in `.cproject`) and source exclusions must be updated — this is done through ADS IDE, which regenerates the makefiles.
- All memory-mapped I/O uses `volatile` semantics (`--language=+volatile` compiler flag).
- CSA (Context Save Area) is hardware-managed on TriCore — each core needs a pre-allocated CSA region in its DSPR for interrupt/call context switching.
